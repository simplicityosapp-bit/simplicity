# Page Builder — תוכנית בנייה (סימפליסיטי)

> מסמך תכנון. נכון ל-2026-06-25. שלב: תכנון בלבד, טרם מימוש.
> מטרה: הרחבת סקשן דפי הנחיתה לבנאי דפים מלא — מרובה-סקשנים, גמיש עיצובית, עם
> חיבור להשארת פרטים ולתיאום פגישות. בנייה גדולה, רב-שלבית.

---

## 1. החזון

מרכז (hub) אחד בניווט שממנו בונים שלושה סוגי דפים, כולם על אותו **מנוע עיצוב משותף**:

| סוג | ייחודי לו | משותף |
|---|---|---|
| **דף נחיתה** | חופש מלא; יכול להכיל בלוק-טופס ו-CTA לפגישות | מנוע הבלוקים + עיצוב + גרירה + רספונסיב |
| **השארת פרטים** | בלוק-טופס במרכז, שדות מותאמים, אישור-ידני, תודה/הפניה | אותו מנוע |
| **תיאום תורים** | סוגי פגישה + זמינות + חלונות זמן + אנטי-דאבל-בוקינג | אותו מנוע |

עקרון-על: **מנוע אחד, שלושה בנאים.** כל סוג הוא בנאי נפרד עם הפיצ'רים המותאמים לו,
אבל הסקשנים, הגרירה, ארגז-העיצוב, והרספונסיב — משותפים.

## 2. החלטות שאושרו (2026-06-25)

| # | החלטה | נבחר |
|---|---|---|
| 1 | מודל עריכה | **סקשנים מובנים** (סטאק אנכי + שורות/עמודות, גמישות בתוך הסקשן). לא קנבס אבסולוטי — כדי שמובייל יעבוד ולשמור על RTL/זכוכית. |
| 2 | מבנה דף | **hub אחד, שלושה סוגים** (נחיתה / השארת-פרטים / תיאום-תורים), כל אחד בנאי נפרד על מנוע משותף. |
| 3 | גישת בנייה | **מאפס (hand-rolled)** — תואם לעקרון "אפס ספריות UI חיצוניות" של הקודבייס. לא Puck. |
| 4 | היקף סבב ראשון | **MVP ממוקד** — מנוע + ~9 בלוקים + העלאת תמונות + גרירה + דסקטופ/מובייל + חיבור לטופס ולפגישות. |

| 6 | פונטים ב-MVP | Heebo + 2-3 פונטים עבריים נבחרים בלבד. פיקר-פונטים מלא → פאזה 4. |

| 8 | רקעים | 10 תמונות הטבע הקיימות **+** העלאת רקע אישי. |
| 9 | מיגרציית דפים ישנים | המרה אוטומטית למבנה-בלוקים בפאזות 2-3, שמירה מלאה של הקיים, הבנאי הישן פעיל עד אימות. |

## 3. הארכיטקטורה

### מודל נתונים
טבלה חדשה `pages` (שם סופי לאישור):
```
pages (
  id, user_id,
  kind        text  -- 'landing' | 'lead' | 'booking'
  title, published, slug,
  theme       jsonb -- { font, palette, background, ... }
  sections    jsonb -- [{ id, type, props, style }]
  -- שדות ספציפיים לסוג (זמינות/סוגי-פגישה לתורים, auto_approve לטופס) — לשקול
  --   האם כ-top-level columns או בתוך theme/props.
  project_id, created_at, updated_at, deleted_at
)
```
ה-JSONB הגמיש כבר מוכח בקוד הקיים (`lead_pages.content`/`fields`,
`booking_pages.content`/`availability`).

### מנוע הבלוקים
- כל סוג בלוק = `{ type, schema, EditForm (נגזר מה-schema), Render }`.
- **renderer טהור אחד** משרת גם תצוגה מקדימה בעורך וגם דף ציבורי → עקביות + פחות קוד.
- מודל גרירה: dnd לסידור סקשנים ואלמנטים בתוך סקשן. **לא** מיקום אבסולוטי.

### ספריית סקשנים ל-MVP (~9)
Hero (כותרת/תת-כותרת/רקע/CTA) · טקסט עשיר · תמונה · אייקון+טקסט (יתרונות) ·
המלצה/ציטוט · כפתור CTA (קישור חיצוני / גלילה לטופס / קביעת פגישה) ·
בלוק-טופס · בלוק-פגישות · מפריד/ריווח.

### עיצוב ורספונסיב
- ערכת-נושא ברמת הדף (פונט, פלטה, רקע) נגזרת מטוקני Mångata הקיימים +
  override פר-סקשן (רקע, ריווח, יישור).
- עיצוב אחד שמסתגל אוטומטית + מתג תצוגה דסקטופ/מובייל בעורך.
- שמירה על שפת הזכוכית (`--mg-card-*`, `backdrop-filter: blur`) ו-RTL.

### אבטחה (נשמר מהמערכת הקיימת)
- Edge Functions: `user_id` מהשרת ולא מהלקוח, honeypot, rate-limit, ולידציית
  שדות סמכותית בשרת (השרת קורא הגדרת שדות מהקונפיג).
- אנטי-דאבל-בוקינג ברמת DB (EXCLUDE constraint) לבלוק-הפגישות.
- העלאת תמונות: bucket ציבורי-לקריאה, כתיבה רק לבעלים, הגבלת MIME+גודל.

## 4. פיצול לפאזות

הסדר מנהל סיכון: ערך חדש (דף נחיתה) נבנה **תוספתי בלי לגעת בדאטה חי**;
המרת המערכות הקיימות נדחית לשלב מאוחר וזהיר.

- **פאזה 0 — תשתית המנוע (בלי UI גלוי):** סכמת `pages`, renderer טהור, מודל
  הבלוקים, תשתית העלאת תמונות (Storage bucket). לא נוגעת ב-`lead_pages`/
  `booking_pages` הקיימות.
- **פאזה 1 — בנאי דף הנחיתה (הלב):** hub בניווט + בנאי סקשנים מלא + גרירה +
  ארגז עיצוב (פונט בסיסי, פלטה, רקע/תמונה) + מתג דסקטופ/מובייל + בלוק-טופס +
  בלוק-CTA-לפגישות + דף ציבורי `/p/<slug>`. **כאן הערך הגדול.**
- **פאזה 2 — קיפול "השארת פרטים" למנוע:** העברת יכולות דפי הנחיתה הישנים אל
  המנוע + מיגרציה שממירה כל `lead_pages` קיים למבנה-בלוקים. הבנאי הישן נשאר
  פעיל עד אימות ההמרה.
- **פאזה 3 — קיפול "תיאום תורים" למנוע:** אותו דבר ל-`booking_pages` (זמינות/
  סוגי-פגישה כ-props של בלוק-הפגישות).
- **פאזה 4+ (אחרי MVP):** פונטים מרובים · ספריית תבניות מוכנות · דראפט/פרסום
  (versioning) · בלוקים-לשימוש-חוזר · FAQ/וידאו/גלריה.

## 5. בעיות וסיכונים מזוהים

1. **גמישות מול מובייל** — נפתר ע"י בחירת סקשנים מובנים (לא קנבס אבסולוטי).
2. **שתי מערכות נפרדות (lead/booking)** — נפתר ע"י hub + מנוע משותף; איחוד
   הדרגתי בפאזות 2-3.
3. **תשתית נכסים חסרה** — מטופל בפאזה 0 (Storage bucket).
4. **פונטים עבריים מוגבלים** — MVP עם רשימה אצורה קצרה; הרחבה בפאזה 4.
5. **דראפט מול פרסום** — נדחה לפאזה 4; ב-MVP פרסום ישיר (סיכון: עריכת דף חי
   נראית למבקרים). לשקול הקפאה בסיסית.
6. **רינדור ציבורי לאנונימיים** — renderer טהור, באנדל קל, SEO.
7. **אבטחת שליחה בטופס גמיש** — ולידציה סמכותית בשרת.
8. **שמירת דאטה קיים** — מיגרציות המרה בפאזות 2-3, בנאי ישן פעיל עד אימות.
9. **Undo/Redo + autosave** — מורכבות ניהול-מצב; להגדיר ב-MVP מינימום (autosave +
   "שינויים לא שמורים").
10. **היקף ענק** — פיצול אגרסיבי, ערך-ביניים שמיש בכל פאזה.

## 6. למה למדנו מהשוק

מ-Carrd / Framer / Webflow / Unbounce / Instapage / Puck / Gutenberg:
- מודל מבוסס בלוקים/סקשנים הוא הסטנדרט המוחלט (לא קנבס חופשי).
- ספריית תבניות מוכנות = ההבדל בין כלי שמשתמשים בו לכלי שנוטשים (פאזה 4).
- בלוקים לשימוש חוזר (Instablocks) — ערך גבוה, פאזה 4.
- בנאים מקצועיים (Webflow/Framer) עובדים במודל תיבות מובנה, לא מיקום אבסולוטי —
  בדיוק הסיבה שדפי Carrd שבירים במובייל.

## 7. נקודות פתוחות לפני פאזה 0

- שם טבלה סופי (`pages` מול `site_pages`).
- האם שדות ספציפיים-לסוג (זמינות, auto_approve) יושבים כ-columns או בתוך JSONB.
- בחירת 2-3 הפונטים העבריים ל-MVP.
- מדיניות גודל/פורמט תמונות + תקרת אחסון לכל משתמש.
