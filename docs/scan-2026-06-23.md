## 🟥 Money path — needs your call (highest priority)

**A. `paymentPlans.js` `addMonths` — installment due-dates wrong for month-end starts (CRITICAL).**
A plan whose 1st due date is the 29th/30th/31st generates broken dates because JS `Date`
overflows short months. Verified: start `2026-01-31`, 4 monthly installments →
`Jan 31, **Mar 3, Mar 31, May 1**` — February skipped, March doubled, April missing.
*Proposed fix:* clamp each installment to the month's last valid day, exactly like the existing
`recurring.js monthlyDateFor` and `reminders.js` already do (Jan 31 → Feb 28 → Mar 31 → Apr 30).
I'll add a test. **Decision:** OK to apply the clamp-to-month-end behavior? (Real bug; left for
you only because it changes generated due dates in the payments feature.)

**B. `usePaymentPlans.markReceived` — can double-count income.**
Marking an installment received creates the income transaction *first*, then flips the
installment. If the flip fails after the tx is created: orphan income tx **and** the installment
still shows unpaid → user retries → **second income transaction** (income counted twice).
*Proposed fix:* compensating-delete of the tx if the flip throws. **Decision:** approve the approach?

**C. `usePaymentPlans.removePlan` — can orphan installments on partial failure.**
Deletes the plan, then updates installments without checking that 2nd call's error; cache already
removed them optimistically, so a failure diverges until a full refetch.
*Proposed fix:* check the error + resync both caches. **Decision:** approve?

**D. `generateInstallments` — no guard for total ≤ 0.** A negative total yields negative
installments and corrupts the plan balance. *Decision:* confirm negatives are never legitimate →
clamp to ≥0 / reject ≤0?

**E. `transactions.amount` has no DB-level CHECK.** Integrity relies entirely on the JS guard
(`assertValidAmount`). A direct PostgREST write (anon key + valid session) could insert a
negative/absurd amount. *Proposed:* add `CHECK (amount >= 0 AND amount <= 1e12)` — but FIRST
audit existing rows for legitimate negatives (refunds?). **Decision:** run the audit query
together in the morning, then decide.

**F. `payment_installments` missing `UNIQUE(plan_id, num)`.** A retried installment-insert against
an already-created plan could duplicate installment numbers (same class the migration 0028 race
fixed for transactions). *Proposed:* additive partial unique index (new migration). I can write
it; needs manual run + a quick "no existing dupes" check. **Decision:** approve?

## 🟨 Performance — improvements, your call on scope

`useGoalEntries`, `useClientStatuses`, `useLeadStatuses`, `useLeadSources`. They use raw
`useState/useEffect`, so multiple widgets on one screen each fire a **duplicate** fetch (home fires

exactly the fan-out React Query was adopted to kill. *Proposed:* migrate to the existing pattern
(template = `useSessions`/`useClients`). Mechanical but touches many consumers — I'd convert one,
verify, then the rest. **Decision:** approve?

**I. Clients screen — O(clients × transactions) balance recompute.** `clientBalance` runs per-client
inside the sort comparator + filters + hero + every card, re-running on every search keystroke.
Quadratic; degrades as a coach's data grows. *Proposed:* precompute a `balanceByClient` map once per
data change and thread it through. Real refactor (touches `ClientCard` props). **Decision:** approve?
